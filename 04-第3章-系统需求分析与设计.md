第 3 章 系统需求分析与设计

3.1 可行性分析

3.1.1 技术可行性

系统采用 B/S 架构与前后端分离模式，后端基于 Spring Boot 与 MyBatis‑Plus，前端基于 Vue.js，数据层使用 MySQL，相关技术成熟稳定。系统通过角色登录与权限控制实现管理员与学生的访问隔离，具备实现招生业务流程的技术可行性。

3.1.2 经济可行性

系统采用开源技术栈，开发成本集中在业务分析与工程实现阶段，硬件与软件投入可控。系统上线后能够减少人工统计与线下沟通成本，提高招生管理效率，具备良好的投入产出比。

3.1.3 操作可行性

系统功能覆盖招生公告、专业信息、报名管理与录取通知等核心环节，流程清晰，操作路径符合管理端与学生端的使用习惯。用户通过浏览器即可完成操作，具备良好的可用性。

3.2 系统功能需求

3.2.1 用户主要功能

学生用户需具备查看招生公告、浏览专业信息、提交报名资料与查询录取通知等功能，并能够在个人中心维护基础信息。系统应提供报名进度反馈与通知提醒，确保学生能够及时了解招生结果。

3.2.2 管理员主要功能

管理员需具备公告发布、专业信息维护、报名审核、录取通知管理、学生信息管理与系统配置功能。管理员还需对文件上传与数据导出进行管理，并通过系统统计了解招生进度与结构变化。

3.3 非功能性需求分析

3.3.1 性能需求

系统需支持并发访问场景，确保公告列表、报名提交与录取查询等核心功能响应稳定。通过分页与缓存策略降低数据库压力，保证系统在报名高峰期保持可用性。

3.3.2 安全性需求

系统需要具备身份认证与权限控制机制，防止未授权访问。关键操作应记录日志以便追踪，报名信息与学生数据需进行安全存储与访问控制。

3.3.3 易用性与可维护性需求

系统界面应简洁直观，便于学生快速完成报名流程。后台模块应具备清晰的菜单结构与操作提示。后端采用分层结构与统一异常处理机制，便于系统维护与后续扩展。

3.4 系统用户用例分析

3.4.1 学生用例图

学生通过登录进入系统，查看公告与专业信息，提交报名资料，并在录取结果发布后查询录取通知。学生可在个人中心修改个人信息并查看历史报名记录。

3.4.2 管理员用例图

管理员登录后发布招生公告，维护专业信息，审核报名信息，生成录取通知，并管理学生信息与系统配置。管理员可通过统计模块查看招生数据。

3.4.3 录取通知用例图

录取通知环节由管理员发布通知内容，系统将通知与学生信息关联，学生可在个人中心查看录取结果，形成报名与通知的闭环流程。
